Evidence: Secret anti-pattern scan of docs and configs
=======================================================

Scan criteria: searched docs/ and configs/ for patterns that would
instruct placing raw rg_ tokens, upstream root bearer, Telegram bot token,
panel session secret, or session signing key into frontend/browser environment.

Patterns checked:
- rg_ (raw token prefix)
- root bearer / bot token / session secret / pepper
- frontend env / browser storage / localStorage / sessionStorage
- window. / REACT_APP / VITE_ / NEXT_PUBLIC / .env. / env.local / public config

Results:
- configs/remnaguard.panel-facade.example.yaml and configs/remnaguard.example.yaml
  contain only env var name references (e.g., bearer_env, bot_token_env,
  secret_env, pepper_env), never raw secret values.

- docs/deployment.md mentions rg_ only in warnings:
  * "Raw rg_... values are rejected" (actor mapping rule)
  * "The browser receives only a panel_ session token... It never receives
     raw rg_... tokens, the upstream root bearer, the Telegram bot token,
     or the session signing secret."

- docs/deployment.md explicitly states:
  "Do not place these values in browser storage, frontend environment
   variables, or public config."

No dangerous frontend secret guidance was found.
